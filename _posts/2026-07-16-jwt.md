# 1 What is A JWT
A Jason Web Token is a simple string that contains some data/information, which has been signed using a secret key by the author of the token.

The signature can be used to verify that the contents/data of the token is the original data authored by the token issuer. If the contents of the token is changed, the signature will no longer match the content of the token, rendering the contents of the token invalid. This allows a token issuer to distribute tokens into the wild, knowing that each of *their* tokens has a unique "signature".

![alt text]({{ site.baseurl }}/assets/images/jwt/token_distribution.png)

If someone presents a token to the token issuer, the token issuer can then check the "signature" of the token to see if it was originally issued by them, and if the contents of the token can therefore be trusted.

![alt text]({{ site.baseurl }}/assets/images/jwt/token_present.png)

The algorithms used to generate the token and its signature ensure that the signature cannot be forged, as it is unique for each token based on its contents.

## 1.1 What Does a JWT Look Like?
A **Jason Web Token** (JWT) is a simple striing that containing 3 parts, using `.` as a separation symbol.

```json
"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiYWRtaW4iOnRydWUsImlhdCI6MTUxNjIzOTAyMn0.KMUFsIDTnFmyG3nMiGM6H9FNFUROf3wh7SmqJp-QV30"
```

The three parts can be decoded to get

- **header** - metadata about that token (JSON)
- **body** - data/content (JSON)
- **signature** - used for check token validity (string)

# 2 Generating a JWT
When generating a JWT, we take some data as input, which we define in the body JSON. We then decide how to generate/verify the signature using token metadata, which we define in the header JSON. Finally we choose a secret key with which to encrypt each token (used to generate the signature), this same key is also used to verify incoming tokens.
1. Encode header JSON & body JSON
2. Combine encoded header & body strings into a message string
3. Use HMAC algorithm to generate a signature
4. Combine message & signature into final output string

![alt text]({{ site.baseurl }}/assets/images/jwt/jwt_overview.png)

[jwt.io](https://www.jwt.io/) is a great tool to learn more about JWT's, and play with some examples.

## 2.1 Encoding Header & Body

The first step is to use base64 encoding to encode the `header` and `body` string.

![alt text]({{ site.baseurl }}/assets/images/jwt/encoding.png)

```python
from base64 import urlsafe_b64encode

def encode_string(string:str) -> str:
    """
    Takes a string and returns a base64url encoded string.
    """
    # byte representation using utf-8 encoding
    utf8_bytes_object = string.encode('utf-8')

    # encode byte object using base64 encoding (convert 8-bit bytes to 6-bit bytes)
    base64_bytes_object = urlsafe_b64encode(utf8_bytes_object)

    # decode bytes object to string
    base64_string = base64_bytes_object.decode('utf-8')

    # remove padding before returning
    return base64_string.rstrip("=")
```

## 2.2 Create Signature

The second step is to generate a signature based on the encoded header, encoded body, and secret key. This signature uses the HMAC algorithm to create a unique hash of a message secured by a secret key. To learn more about HMAC see Appendix A and Appendix B.

1. The `encoded_header` and `encoded_body` strings are joined together into a `message`,  using a `.` as separation symbol.

2. The `message` is then passed through the HMAC algorithm using a hash algorithm (e.g. SH256) defined in the header, using the `secret_key` as the hashing key. The result is the `signature`

The HMAC algorithm can be used with many different hashing algorithms, one of the most common ones is SHA-256. To learn more about this hashing algorithm specifically, see [this blog post]({% post_url 2026-07-06-sha-256 %})

![alt text]({{ site.baseurl }}/assets/images/jwt/create_signature.png)

```python
import hmac
import hashlib
from base64 import urlsafe_b64encode

def create_signature(header:str, body:str, secret:str) -> str:
    """
    Takes a header, body, and secret and returns a base64url encoded signature.
    """
    # combine encoded body and encoded header
    message = encode_string(header) + "." + encode_string(body)

    # generate signature
    signature = hmac.new(
        key=secret.encode('utf-8'),
        msg=message.encode('utf-8'),
        digestmod=hashlib.sha256
    ).digest()

    # base64 encode signature
    encoded_signature = urlsafe_b64encode(signature).decode('utf-8').rstrip("=")
    return encoded_signature
```

## 2.3 Create Token

Finally the `encoded_header` string, `encoded_body` string, and `signature` string are joined together (where `.` is again used as separation symbol) to form the `JWT` token.

![alt text]({{ site.baseurl }}/assets/images/jwt/create_token.png)

```python
signature = create_signature(header, body, secret_key)

JWT = encode_string(header) + "." + encode_string(body) + "." + signature
```

# 3 Verifying a JWT

The idea of using a JWT, is that when the user authenticates, the servers sends them a token (JWT) inside of a cookie. The client machine will store this cookie, and whenever the client makes a request to the server, it will send a cookie containing the authentication token along with the request. The server will then verify if the token it received is valid before responding with the data.

The whole point of JWTs is to not save any user information on the server itself. Instead the header and body of the token can be used in combination with the secret_key stored on the server to generate the correct signature. This correct signature can then be compared with the provided signature.

## 3.1 Decode Header and Body

A JWT is a string, which is split into three parts using a `.` as separation symbol. Each substring is base64 encoded, and once decoded will provide data in JSON format:

```python
from base64 import urlsafe_b64decode

def base64url_decode(input:str):
    """
    Takes a base64url encoded string and returns the original string.
    """
    # add padding to input
    padding = '=' * (4 - (len(input) % 4))
    input = input + padding

    # convert string to bytes object
    base64_bytes_object = input.encode('utf-8')

    # decode base64 bytes object to utf-8 encoded string
    utf8_bytes_object = urlsafe_b64decode(base64_bytes_object)

    # decode utf-8 encoded string
    string = utf8_bytes_object.decode('utf-8')

    return string
```

## 3.2 Generate Correct Signature

The correct signature can be found by creating a signature from the `header` and `body` found in the `JWT`.  The generated `correct_signature` should be equal to the `signature` found in the `JWT`.

```python
header, body, signature = JWT.split(".")

header = base64url_decode(header)
body = base64url_decode(body)

correct_signature = create_signature(header, body, secret='bamboozle')

correct_signature == signature
```

The signature generated by `create_signature` will only match the signature found in the JWT if the header and body correspond to the original header and body. If someone takes the JWT, modifies the body, and tries to submit that token, then the expected signature (token header + token body [modified] + server secret) will not equal the token signature (which was computed using token header + token body [original] + server secret)

## Session
Traditionally speaking the authorization is performed by storing an authorized user in the server memory.

The user logs in with their username and password, if the right credentials are provided the server will create a session for the user in memory. As part of the authentication process the server will send a session ID inside of a cookie to the client.

Whenever a user wants more data from the server, it sends a request with a cookie containing the session ID. The server will only return data to the user if the session ID is in the list of active session IDs in the server’s memory.

!Untitled

## Token

Using a JWT is very similar, except that no data is stored in the server memory. Instead only a single Secret Key is necessary to check whether or not the provided JWT is correct or not.

!Untitled

Similar to a session, the user logs in using their username and password credentials. The server then creates a Jason Web Token, which it sends to the client inside of a cookie.

**Advantages**

The main advantage is that the authorization data is not stored server-side, but client-side. For each request the server needs to check whether the client is a valid user, instead of checking an ID against a list of valid IDs, it checks whether the token is valid. To verify if the token is valid, only the secret key is required.

**Use Case**

Not only will this improve the server performance (no need to keep an active session ID for each user using the web-app), but also allows easy authorization between different servers.

If there are two servers using sessions, then the two servers would need to constantly synch the list of session IDs for active users. If the two servers are using tokens, then they simply need to use the same secret_key to validate the tokens.
